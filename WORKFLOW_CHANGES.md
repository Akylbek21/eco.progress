# Изменения workflow заявок

## Что изменено

Раздел "Заявки" переведен с общего набора старых статусов на реальный порядок выполнения работы сотрудниками ECOPROGRESS GROUP. Теперь поле `order.status` отражает текущий этап бизнес-процесса, а договор, счет, оплата, экология и лаборатория остаются отдельными рабочими блоками.

## Новые статусы заявки

1. Консультация
2. Анализ
3. КП
4. Договор
5. Счет на оплату
6. Проектирование / Лаборатория / Вывоз / Утилизация
7. Проверка результата
8. Готово
9. Завершено

Дополнительно сохраняется статус `Отменено`.

Единое описание статусов находится в `src/data/mockData.ts` в `orderStatusDefinitions`. Для каждого этапа указаны `id`, `label`, `description`, `order`, `category`, `clientVisibleLabel` и `employeeActionLabel`.

## Переход между этапами

Основная последовательность строится в `src/utils/crm.ts` через `getWorkflowForOrder(order)`.

Кнопка следующего этапа в кабинете сотрудника использует `getNextOrderStatus(order)`. После этапа `Счет на оплату` переход блокируется, пока `paymentStatus` не станет `paid`. Когда счет оплачен, заявка автоматически переходит в рабочий этап по выбранной услуге.

Отдельные автоматические переходы:

- создание заявки: `Консультация`;
- отправка договора и счета: `Счет на оплату`;
- онлайн-оплата или ручная отметка `paid`: рабочий этап услуги;
- завершение проектирования или лаборатории: `Проверка результата`;
- после проверки сотрудник переводит заявку в `Готово`, затем в `Завершено`.

## Определение рабочего этапа

Функция `getWorkStageByService(service)` находится в `src/utils/crm.ts`.

Логика:

- лаборатория, анализы, исследования, пробы, замеры -> `Лаборатория`;
- транспортировка или вывоз -> `Вывоз`;
- экологическое проектирование, разрешения, отчеты, ОВОС, ПДВ, НДВ, ПНООЛР, РООС, ПДС, СЗЗ, ПЭК, ППМ -> `Проектирование`;
- утилизация, переработка, полигон, размещение, захоронение, отходы -> `Утилизация`;
- все остальное -> `Проектирование`.

## Миграция старых статусов

Старые статусы из localStorage безопасно мигрируются в `normalizeOrderStatus()`:

- `Новая` -> `Консультация`;
- `В обработке` и `Ожидает документы` -> `Анализ`;
- `В работе` -> рабочий этап по услуге;
- `На проверке` -> `Проверка результата`;
- `Готово`, `Завершено`, `Отменено` сохраняются.

При чтении localStorage заявки нормализуются и записываются обратно уже в новом формате.

## Как проверить вручную

1. Очистить localStorage или открыть приложение с текущими данными, чтобы проверить миграцию старых заявок.
2. Открыть кабинет сотрудника: `/staff/orders`.
3. Проверить фильтр по статусам: должны отображаться новые этапы.
4. Открыть карточку заявки и проверить timeline:
   `Консультация -> Анализ -> КП -> Договор -> Счет на оплату -> рабочий этап -> Проверка результата -> Готово -> Завершено`.
5. Перевести заявку до этапа `Счет на оплату`.
6. Поставить оплату `Оплачено`; заявка должна перейти:
   - лабораторная услуга -> `Лаборатория`;
   - вывоз отходов -> `Вывоз`;
   - утилизация/полигон -> `Утилизация`;
   - экологическое проектирование -> `Проектирование`.
7. В кабинете клиента проверить, что путь заявки понятен клиенту: рабочий этап показывается как `Выполнение работ`, а в тексте уточняется внутренний тип работ.

## Ревизия годовых заявок 2026-05

- Годовая заявка остается `annual_active` весь срок договора и не закрывается после первого квартала.
- Завершение годовой заявки разрешено только если все 4 квартала `completed`, по каждому кварталу есть результат и нет активного долга.
- Текущий квартал определяется по `periodStart` / `periodEnd` внутри заявки, а не по календарному номеру месяца.
- Для кварталов договора поддержаны `calendar_quarters` и `contract_quarters`.
- Оплата больше не меняет рабочий статус `completed` обратно на `blocked_by_debt`; блокировка долгом применяется только к незавершенным кварталам.
- Финансовые детали в карточке заявки доступны только `ADMIN` и `ACCOUNTANT`; рабочие роли видят краткое состояние оплаты.
